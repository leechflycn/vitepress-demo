# 网络折腾血泪史：透明代理与 407 报错大坑

在折腾局域网全屋科学上网（透明代理）的过程中，我们遇到过几个极其隐蔽的坑，这里复盘分享一下，希望能帮你少走弯路。

## 1. OpenClash 局域网共享的 "407" 刺客

**场景还原**：
为了让家里的所有设备（包括虚拟机里的 Linux 和主控机 Windows）无需安装任何代理软件就能翻墙，我们将设备的网关和 DNS 都指向了旁路由（`192.168.x.x`）。
结果配置完之后，设备不仅没翻出去，连国内网站也打不开了，彻底断网。

**排查过程**：
来来回回测试了很久，最后通过命令行抓包发现，所有流量都被网关拦截，并返回了 `HTTP 407 Proxy Authentication Required` 错误。

**避坑指南**：
如果你在 OpenClash 中开启了“允许局域网连接”，**千万不要在底下的“HTTP 代理端口认证”里填写账号和密码**。一旦开启了认证，局域网内的设备如果没有在请求头里带上密码，流量就会被静默丢弃，导致彻底断网。
解决办法：清空账号密码，让局域网流量无条件通行。

## 2. Linux 虚拟机的网关遗漏症

**场景还原**：
在 PVE 宿主机上开了一台 Ubuntu 虚拟机用来跑 AI 助手（OpenClaw）。旁路由配置好后，Windows 主机上网很爽，但 Ubuntu 里的 AI 助手死活查不了网页，报错请求超时。

**排查过程**：
一开始以为是 Docker 容器网络问题，来回改 Docker 的 DNS。最后发现是 Ubuntu 系统底层的网关没指过去。

**避坑指南**：
在 Linux 环境下，配置透明代理一定要去修改网络配置文件（比如 Ubuntu 的 Netplan 配置：`/etc/netplan/00-installer-config.yaml`）。
确保 `gateway4`（或 `routes`） 指向旁路由，且 `nameservers` 的 `addresses` 也必须是旁路由 IP。修改完必须运行 `sudo netplan apply`，否则系统底层的流量根本不知道该往哪走。
